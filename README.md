# ABYS

ABYS is the destination repository for deployable software infrastructure, product-development intelligence, agent orchestration, memory systems, Codex-compatible build workflows, simulation dashboards, creator tools, workflow engines, and execution scaffolds.

ABYS is not the artifact canon and not the interenterprise protocol layer. It is the execution machine.

## Routing Rule

All ABYS-related work routes here:

- ABYS Forge
- ABYS memory kernel
- ABYS agent orchestration
- ABYS Codex task packets
- ABYS workflow engines
- ABYS simulation dashboards
- ABYS market/product intelligence
- ABYS execution constitution
- intake routing from concept to issue to PR
- tests, CI, deployment, and implementation scaffolds

ITEM-related artifact doctrine must remain in `injammed/ITEM`.

SYNTEL-related A2A / M2M protocol infrastructure must remain in `injammed/SYNTEL`.

## Canon-vs-slop principle

ABYS material must terminate in execution. Strong concepts become task packets, code, schemas, tests, docs, workflows, PRs, deployments, dashboards, or measurable process improvements. Weak concepts are rejected, redirected, or compressed until they produce an executable next action.

## Initial repository structure

```txt
/docs              Execution doctrine, intake routing, operating rules
/schemas           Machine-readable task and workflow schemas
/examples          Example task packets and product-development flows
/src               Reference implementations and tools
/tests             Validation tests for schemas, workflows, and code
.github            Issue templates and CI workflows
```

## Build direction

ABYS should behave like a persistent product-development intelligence system: interpret intent, route concepts to the right repository, decompose tasks, produce Codex-compatible work packets, validate outputs, preserve successful patterns, and recursively improve the workflow.
