# ABYS Intake Routing

ABYS exists to terminate concepts in executable work: issues, PRs, code, schemas, tests, workflows, deployments, dashboards, and measurable process improvements.

A concept is not ABYS-ready until it can be expressed as a task packet with an objective, assumptions, routing decision, files, implementation steps, validation, risks, terminal output, and next recursive improvement.

## Repository Routing

Route work to `injammed/ITEM` when the core output is an artifact record, canon decision, physical/digital/narrative object specification, anti-entropy object doctrine, or canon-vs-slop review.

Route work to `injammed/ABYS` when the core output is execution infrastructure: task packets, deterministic routing, product-development workflows, repo automation, schemas, CI, tests, implementation scaffolds, business pipelines, or Codex-compatible build steps.

Route work to `injammed/SYNTEL` when the core output is signed inter-agent protocol infrastructure: identity, envelope, task contract, verification receipt, audit replay, or human escalation hooks.

## Slop Rejection Rules

Reject or quarantine ABYS work when it is only motivational language, duplicate doctrine, vague mythic material, or a plan without files to create/edit and tests to run.

A valid ABYS task packet must answer:

1. What repository should change?
2. What files should be created, edited, deleted, or tested?
3. What command proves the work is valid?
4. What terminal output closes the loop?
5. What becomes easier in the next recursive cycle?

## Current v0 Primitive

The canonical v0 ABYS task-packet primitive is:

```txt
schemas/task-packet.schema.json
```

Use it before opening implementation work so concepts arrive as bounded execution packets rather than prose sprawl.

## Handoff Discipline

ABYS may reference ITEM and SYNTEL, but it should not absorb their domain content. ABYS should produce the queue, task packet, workflow, or validation harness that lets the correct repository receive the right work.
