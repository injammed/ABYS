# ITEM / ABYS / SYNTEL Autonomous Tasking Hierarchy

## Hierarchy

```txt
1. ITEM
2. ABYS
3. SYNTEL
```

## Purpose

This document updates ABYS autonomous tasking so that ITEM, ABYS, and SYNTEL are treated as distinct but mutually feeding systems.

## System Roles

### ITEM

ITEM creates cultural and symbolic demand:

- canon
- Full-Mode Artifacts
- AETIMM / SLATRA
- public curation
- Golden Dust / contribution reputation
- future civilization archive

### ABYS

ABYS converts concepts into deployable infrastructure:

- execution packets
- orchestration
- workflow engines
- persistent operational memory
- verification loops
- Codex handoff
- monetizable software products

### SYNTEL

SYNTEL receives the residual protocol layer:

- agent-to-agent communication
- machine-to-machine messaging
- interenterprise coordination
- signed envelopes
- capability discovery
- task contracts
- verification receipts
- audit trails

## Feed Loop

```txt
ITEM → ABYS
Canon needs become product/infrastructure requirements.

ABYS → SYNTEL
Execution systems reveal protocol and inter-agent communication needs.

SYNTEL → ABYS
Protocols improve orchestration, verification, and cross-tool execution.

ABYS → ITEM
Infrastructure makes the public canon app real.

ITEM → SYNTEL
Artifact provenance and public participation generate protocol needs.

SYNTEL → ITEM
Identity, provenance, and verification protocols make canon more durable.
```

## Task Routing

### Route to ITEM when task concerns:

- AETIMM / SLATRA
- ITEM canon
- Full-Mode Artifacts
- public museum / registry
- Golden Dust
- canon gate
- symbolic artifact economics
- anti-slop doctrine

### Route to ABYS when task concerns:

- deployable app/software infrastructure
- execution workflows
- Codex task packets
- agent orchestration
- memory systems
- verification systems
- market/product automation
- boring friction / logistics / billing workflows

### Route to SYNTEL when task concerns:

- M2M / A2A communication
- agent identity and capability discovery
- signed message envelopes
- task contract formats
- verification receipt formats
- interenterprise agent coordination
- cross-system protocols
- residual synthesis between ITEM and ABYS

## Autonomous Tasking Rule

Every autonomous task should classify itself into one of:

```txt
ITEM
ABYS
SYNTEL
split
hold_for_review
```

If split:

- ITEM receives canon / symbolic / cultural layer.
- ABYS receives executable infrastructure layer.
- SYNTEL receives protocol / interoperation layer.

## North-Star Compression

```txt
ITEM canonizes meaning.
ABYS operationalizes execution.
SYNTEL standardizes communication.
```

Together:

```txt
Meaning → Execution → Communication → Feedback → New Meaning
```
