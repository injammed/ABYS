# GPT → Digest → Codex → Repository Handoff Pipeline

## Objective

Automate a selective delivery sequence where code and product content generated in ChatGPT is compressed into Codex-digestible work packets before it reaches the canonical destination repositories.

Canonical destinations:

- `injammed/ABYS` for ABYS infrastructure, execution systems, agent orchestration, memory, workflows, market/product intelligence, dashboards, and deployable software.
- `injammed/ITEM` for ITEM, Aeternum Immutablis, Animentvm, symbolic continuity, semantic object graphs, provenance, and mythic-economic artifact systems.

## Core Rule

Do not send raw chat directly to repositories.

Always route through this sequence:

```txt
ChatGPT concept / code content
→ intake classifier
→ compression digest
→ destination router
→ Codex task packet
→ Codex execution / review
→ repository PR or commit
→ verification result
→ memory update
```

## Why This Exists

Raw chat is high entropy. Repositories require low-entropy executable structure. Codex works best when it receives scoped tasks with context, files, constraints, and acceptance tests.

## Pipeline Stages

### 1. Intake Classifier

Classifies incoming content.

```yaml
content_type:
  - concept
  - architecture
  - code
  - documentation
  - schema
  - workflow
  - test
  - market_intelligence
  - symbolic_canon

destination:
  - ABYS
  - ITEM
  - both_split
  - hold_for_review
```

### 2. Compression Digest

Compresses conversational material into operational structure.

Required fields:

```yaml
objective: string
source_summary: string
preserve:
  - string
discard:
  - string
assumptions:
  - string
open_questions:
  - string
```

### 3. Destination Router

Routing rules:

```yaml
ABYS:
  repository: injammed/ABYS
  receives:
    - software infrastructure
    - agent orchestration
    - GPT to Codex workflows
    - memory systems
    - workflow engines
    - dashboard scaffolds
    - market/product analysis engines
    - executable architecture

ITEM:
  repository: injammed/ITEM
  receives:
    - ITEM canon
    - Aeternum Immutablis
    - Animentvm
    - symbolic object continuity
    - semantic object graphs
    - provenance systems
    - mythic-economic artifacts

both_split:
  rule: split into two packets, never duplicate raw content
```

### 4. Codex Task Packet Generator

Every implementation task must become a packet with:

```yaml
task_id: string
destination_repo: string
title: string
objective: string
context: string
files_to_create_or_modify:
  - path: string
    purpose: string
acceptance_tests:
  - string
constraints:
  - string
risk_notes:
  - string
verification_plan:
  - string
```

### 5. Codex Execution Gate

Codex should receive only task packets that satisfy:

- bounded scope
- clear repository destination
- explicit files
- acceptance tests
- constraints
- rollback or review path

### 6. Repository Delivery

Preferred delivery modes:

1. GitHub issue with Codex task packet.
2. Branch + PR generated by Codex.
3. Direct commit only for small documentation or routing changes.

## Example Packet

```yaml
task_id: abys-codex-0001
destination_repo: injammed/ABYS
title: Implement ABYS digest queue schema
objective: Add a schema for storing compressed GPT-to-Codex task packets before execution.
context: Raw chat should not be committed directly. The digest queue stores structured work packets.
files_to_create_or_modify:
  - path: supabase/migrations/0002_codex_digest_queue.sql
    purpose: Create digest_queue table.
  - path: lib/digest/schema.ts
    purpose: Define TypeScript types for digest packets.
acceptance_tests:
  - Migration creates digest_queue table.
  - Packet status supports queued, sent_to_codex, executed, rejected, verified.
  - destination_repo must be either injammed/ABYS or injammed/ITEM.
constraints:
  - Do not mix ITEM canon with ABYS infrastructure.
  - Do not execute unreviewed high-impact operations automatically.
risk_notes:
  - Bad routing could pollute the wrong repository.
verification_plan:
  - Create sample ABYS packet and ITEM packet.
  - Confirm each resolves to correct destination.
```

## Queue Status Lifecycle

```txt
captured
→ digested
→ routed
→ codex_ready
→ sent_to_codex
→ executed
→ verified
→ archived
```

Failure states:

```txt
rejected
blocked
needs_human_review
misrouted
```

## Anti-Delusion Constraint

A task is not real until it produces at least one of:

- issue
- branch
- PR
- commit
- passing test
- deployed service
- verified documentation artifact
