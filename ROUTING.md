# Repository Routing Policy

## Canonical Destinations

- `injammed/ABYS` receives ABYS work.
- `injammed/ITEM` receives ITEM work.

## ABYS Scope

Route to `injammed/ABYS` when the work concerns:

- ABYS
- deployable product infrastructure
- GPT to Codex execution loops
- autonomous incubation council outputs
- agent orchestration
- persistent operational memory
- workflow engines
- market/product intelligence
- simulation dashboards
- creator/product tools
- software architecture and implementation scaffolds

## ITEM Scope

Route to `injammed/ITEM` when the work concerns:

- ITEM
- Aeternum Immutablis
- Animentvm
- symbolic object continuity
- semantic object graphs
- ITEM Vault
- ITEM Continuity Engine
- mythic-economic artifact systems
- AI-native provenance for symbolic assets

## Conflict Rule

If a task mentions both ITEM and ABYS:

1. Put executable software infrastructure, build systems, Codex workflows, and product orchestration in ABYS.
2. Put symbolic continuity doctrine, ITEM object schemas, mythology, provenance, and economic framing in ITEM.
3. Cross-link rather than duplicate when possible.

## Anti-Contamination Rule

Do not write ABYS execution infrastructure into ITEM unless explicitly requested.
Do not write ITEM symbolic canon into ABYS unless it is needed as product content, fixture data, or integration reference.
